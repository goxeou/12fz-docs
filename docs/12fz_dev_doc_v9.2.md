### 2.5 数据库架构（统一）

**一个PG库，三个Schema：**

```
postgresql://PG:5432/12fz
├── biz      ← 业务数据（订单/库存/商品/财务）
├── chat     ← 聊天数据（消息/群组/好友关系）
└── sys      ← 系统数据（用户/角色/权限/认证）
```

- 所有模块共享同一PG实例，同一套用户体系
- 不同Schema逻辑隔离，但SQL可以跨Schema Join
- AI层通过Go API读写数据，不直接操作PG
- 聊天消息存PG（和业务数据同库，方便AI查上下文）

### 2.7 统一认证

插件式OAuth Provider，一套接口支持所有登录方式：

| 方式 | 优先级 | 实现 |
|:----|:----:|:----:|
| 邮箱+密码 | 🔴 P0 | bcrypt |
| 手机+验证码 | 🔴 P0 | 聚合API按地区切换 |
| Google OAuth | 🔴 P0 | golang.org/x/oauth2 |
| Apple ID | 🔴 P0 | Sign in with Apple |
| 微信扫码 | 🔴 P0 | 微信开放平台 |
| 预留扩展 | 🔵 预留 | 写一个Provider实现 |

**核心接口：**
```go
type OAuthProvider interface {
    Name() string
    AuthURL(state string) string
    Callback(code string) (*OAuthUser, error)
}
```

加一个新登录方式 = 写一个Go文件 + 一行import。

## 第五部分：统一认证体系详细设计

### 5.1 核心数据模型

```sql
-- 统一用户表
CREATE TABLE unified_users (
    id              BIGSERIAL PRIMARY KEY,
    email           VARCHAR(255) UNIQUE,
    phone           VARCHAR(20) UNIQUE,
    phone_country   VARCHAR(4),
    password        VARCHAR(255),        -- bcrypt hash
    display_name    VARCHAR(100),
    avatar_url      TEXT,
    locale          VARCHAR(10) DEFAULT 'zh-CN',
    timezone        VARCHAR(50) DEFAULT 'Asia/Shanghai',
    status          VARCHAR(20) DEFAULT 'active',
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- OAuth绑定（可绑定多个）
CREATE TABLE unified_oauth_bindings (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT REFERENCES unified_users(id) ON DELETE CASCADE,
    provider        VARCHAR(32) NOT NULL,
    provider_id     VARCHAR(128) NOT NULL,
    provider_email  VARCHAR(255),
    provider_name   VARCHAR(100),
    raw_data        JSONB,
    bound_at        TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (provider, provider_id)
);

-- 角色表
CREATE TABLE unified_roles (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(50) UNIQUE,
    description TEXT
);

-- 用户-角色关联
CREATE TABLE unified_user_roles (
    user_id     BIGINT REFERENCES unified_users(id),
    role_id     INT REFERENCES unified_roles(id),
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (user_id, role_id)
);
```

### 5.2 JWT方案

- **Access Token**：15分钟有效期，短时效
- **Refresh Token**：30天有效期，持久化
- Token在Redis中做黑名单（登出时加入）

### 5.3 国际化登录页

按用户IP/浏览器语言/历史登录方式动态排序：
- 中国用户：微信 > 手机 > 邮箱 > Google > 其他
- 美国用户：Google > Apple > 邮箱 > Instagram > 其他
- 日本用户：LINE > Google > Apple > 邮箱 > 其他

## 第六部分：技术规范（红线）

### 6.1 数据库规范
- 所有表必须带 `created_at` / `updated_at` 字段
- 数据库变更必带回滚脚本
- 严禁Python/Agent直接拼SQL操作业务DB
- 跨语言调用必须经过接口契约（OpenAPI/gRPC）
- MySQL仅存量运行，不新建业务表

### 6.2 安全规范
- API Key分环境（testnet/prod），`$HOME/.keys/`下600权限，禁止入Git
- 交易订单签字逻辑独立成Go library，Python/Agent不得直接组装订单payload
- 数据库凭证使用环境变量或Vault注入，禁止明文配置
- Agent执行敏感操作需在沙箱内运行

### 6.3 部署规范
- 开发站验证 → 老板确认 → 同步生产（绝对红线）
- **生产站绝对禁改**，没有老板明确指令不得触碰
- **15天静默期（2026-06-02 起）：** 15天内不考虑生产站，等老板通知再同步生产（老板确认）
- 改前备份（`cp xxx xxx.bak.$(date +%Y%m%d_%H%M%S)`）
- 改完先验证再重启，不盲动

### 6.4 开发规范
- Go服务提供 pprof 端点，供Agent动态采样
- Python模型服务强制使用 model_offload 策略（GPU/CPU自动切换）
- Java堆内存与堆外内存分离，防OOM
- PHP仅用于无状态前端胶水层，禁止长任务

| 模块 | 语言 | 理由 |
|------|------|------|

3. 用户体系统一 → 共用 unified_users
4. AI只能读数据、通过API执行业务操作，不能直接写业务表

### 10.9 客服系统详细设计（老板确认 2026-06-04）

聊天系统作为客服系统使用，支持AI初筛和人工转接。

**客服流程：**

```
客户在商城/订单页
   ↓ 点击"联系客服"
AI助手初筛
   ├─ 简单问题 → AI直接回答 ✅
   └─ 复杂问题 → 转人工客服
                     ↓
            客服A接单
             ├─ 能处理 → 解决 ✅
             └─ 不能处理 → 点"转接"
                              ├─ 转客服B（一对一）
                              ├─ 转主管
                              └─ 转技术部门（可多人）
                              ↓
                      带完整上下文自动转交
                      客户不需要重复描述
```

**转接设计要点：**
1. **转接带上下文** — 对话历史、订单号、客户信息、问题类型，全部自动带过去
2. **转接记录** — 每次转接记录到日志（谁转给谁、原因、时间），可追溯
3. **转接到个人** — 转给指定客服B/主管
4. **转接到客服组** — 转给整个客服组（如技术组、仓储组），组内任意在线客服接单
5. **AI初筛前置** — 客户消息先过AI，AI搞不定再转人工，减少客服压力
6. **客服状态** — 在线/离线/忙碌，只能分配给在线客服
7. **客服分组** — 不同业务线不同客服组（售前组/售后组/技术组/仓储组）

**与业务页面衔接：**
- 订单详情页 → "联系客服" → 自动带入订单号
- 库存页 → "通知采购" → 自动带入物料信息
- 售后页 → "申请售后" → 创建客服工单

**技术实现：**
- 客服状态管理和分配逻辑在Go聊天服务中实现
- AI初筛引擎在Python AI层
- 转接记录存PG（chat schema），与聊天记录关联

### 10.10 AI Agent集成方案（2026-06-04新增）

**核心理念：** 开发团队现有的Hermes Agent（chaogu-ai/gong3/服务器技术/高级工程师）应成为新聊天系统的原生用户，替代当前飞书群聊+中继的割裂模式。

#### 10.10.1 当前模式（过渡期）

```
飞书群聊 ←→ 4个Hermes Agent ←→ 中继（仅限于心跳+手动推送）
```

问题：群聊、Agent、中继三套东西割裂，Agent看不到群聊上下文，依赖中继手工转发。

#### 10.10.2 未来模式（聊天系统上线后）

```
┌─────────────────────────────────────┐
│       新聊天系统（Go + WebSocket）    │
│                                      │
│  老板 ←→ chaogu-ai Agent             │
│  gong3 Agent ←→ 服务器技术 Agent      │
│  高级工程师 Agent ←→ 客户（通过AI客服） │
│  业务系统告警 → 自动推送到相关Agent     │
└─────────────────────────────────────┘
```

- **没有中继** — Agent和人类在同一个聊天系统里
- **没有飞书** — 内部沟通全部在新聊天系统完成
- **Agent全功能** — 收消息、读历史、主动推送、@别人，全部原生态

#### 10.10.3 Agent账号体系

每个Agent作为一个特殊用户，以 `agent/xxx` 格式接入统一用户系统：

| Agent | 账号 | 角色 |
|-------|------|------|
| chaogu-ai | `agent/chaogu-ai` | 团队协调、代码审核、交易信号执行 |
| gong3 | `agent/gong3` | Go后端开发、架构设计 |

| AI客服 | `agent/kefu` | 客户自动回复（未来） |

Agent账号属于 `unified_users` 表，类型标记为 `agent`，走统一JWT认证。

#### 10.10.4 技术实现

1. **Agent = WebSocket客户端** — 每个Agent通过Hermes Agent平台连接聊天系统WebSocket
2. **标准消息协议** — Agent发送/接收的消息格式与普通用户完全一致
3. **主动推送接口** — 聊天系统提供REST API，供业务系统直接推消息给Agent（如：库存告警→@agent/server-tech）
4. **审计日志** — 所有Agent操作、消息、推送记录到chat schema
5. **无需中继** — Agent直接读写聊天系统的消息，不需要中间转发层

#### 10.10.5 Agent能力矩阵
